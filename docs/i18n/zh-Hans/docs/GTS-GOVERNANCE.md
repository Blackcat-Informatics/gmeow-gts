# GTS 治理与发布策略

> [`docs/GTS-GOVERNANCE.md`](../../../../docs/GTS-GOVERNANCE.md) 的信息性中文翻译。英文文档仍然是治理、安全、发布、许可、贡献、行为义务、披露流程和可执行命令的权威来源。本翻译遵循 [`docs/i18n/GLOSSARY.md`](../../GLOSSARY.md)，仅供参考。

本文件定义了 GTS 的轻量级变更控制流程、扩展注册策略、兼容性合约以及 v1.0 发布候选路线。它是对 [`GTS-SPEC.md`](./GTS-SPEC.md) 中的有线格式规范、[`GTS-CONFORMANCE.md`](./GTS-CONFORMANCE.md) 中的一致性策略、[`GTS-SECURITY-POLICY.md`](./GTS-SECURITY-POLICY.md) 中的配置文件/安全策略以及 [`../SECURITY.md`](../SECURITY.md) 中的仓库安全披露策略的补充。

GTS 与本体无关。GMEOW 是主要的下游使用者和分发用例，但 GTS 读取器和写入器不需要 GMEOW 词汇表、工具或语义。

## 1. 变更类别

GTS 对核心格式变更以及配置文件 (profile) 或注册表添加采用不同的治理路径。
其目标是在允许配置文件和部署模式演进的同时，保持线缆格式窄腰结构的稳定性。

| 变更类别 | 示例 | 所需路径 |
|---|---|---|
| 核心线缆格式变更 | 标头或帧语法、确定性 CBOR 规则、id/哈希原像、`prev` 语义、段边界、转换目录机制、核心折叠语义。 | GTS 改进提案 (GIP)、规范 PR、一致性语料库影响审查以及跨引擎实现计划。 |
| 读取器/写入器一致性变更 | 基准读取器层级、诊断行为、向量预期、写入器确定性要求。 | 行为发生变化时使用 GIP；对于保留行为的澄清使用文档/语料库 PR。 |
| 可选标准配置文件 | `files`、`evidence`、`opaque`、`bundle`、`stream` 或旨在由 GTS 核心维护的新配置文件。 | 配置文件提案，如果增加了标准义务、安全策略或一致性层级，则需附加 GIP。 |
| 特定领域配置文件 | GMEOW 分发配置文件、`music-package`、`agent-memory` 以及第三方应用配置文件。 | 配置文件注册审查；除非该配置文件需要新的核心行为，否则无需核心 GIP。 |
| 注册表添加 | 新编解码器名称、诊断代码、帧类型、配置文件名称或转换目标。 | 第 3 节中的注册表政策。 |
| 实现或包变更 | 引擎 API、CLI 人机工程学、打包、文档、示例。 | 除非改变了一致性行为，否则遵循正常的 issue/PR 流程。 |

## 2. GTS 改进提议 (GTS Improvement Proposals)

在更改可以改变核心 GTS 语义或符合标准的实现所要求的行为之前，必须提交一份 GTS 改进提议 (GTS Improvement Proposal, GIP)。

### 2.1 何时需要 GIP

在以下情况下需要提交 GIP：

- 对标头或帧 (frame) 语法的任何更改；
- 对哈希原像、签名原像或确定性 CBOR 要求的任何更改；
- 对段 (segment) 构成、`prev` 链规则或折叠 (fold) 语义的任何更改；
- 增加、删除或重新分类基准读取器 (Reader)、写入器 (Writer)、验证工具或全功能读取器 (Full Reader) 的要求；
- 将可选能力变为强制性；
- 增加由 GTS 核心维护的可选标准配置文件 (profile)；
- 将编解码器、帧类型或诊断信息提升为核心要求的注册条目；
- 更改线缆格式、语料库 (corpus)、包或配置文件 (profile) 的兼容性策略。

对于编辑性澄清、示例、特定领域的配置文件 (profile) 注册、包元数据或保持可观察的一致性 (conformance) 行为的实现重构，不需要提交 GIP。

### 2.2 GIP 形式

GIP 应该 (SHOULD) 以 GitHub issue 或检入的设计文档形式开启，包含：

- 标题和摘要；
- 变更类别和受影响的规范章节；
- 动机和非目标；
- 准确的规范行为变更；
- 对 wire、corpus、程序包和配置文件 (profiles) 的兼容性影响；
- 待添加或更新的一致性向量 (conformance vectors)；
- 受影响的引擎和实现计划；
- 安全和隐私考量；
- 迁移计划和发布里程碑；
- 已考虑的替代方案；
- 决策状态：`draft`、`proposed`、`accepted`、`implemented`、`rejected` 或 `superseded`。

被接受的 GIP 通过正常的拉取请求 (pull requests) 实现。实现 GIP 的 PR 必须 (MUST) 链接该 GIP 并更新相关的规范、一致性语料库 (conformance corpus)、文档和引擎。

## 3. 注册表治理 (Registry Governance)

权威注册表表格可能存在于规范、一致性文档、安全策略或未来的 `docs/registries/` 文件中。本节负责每个注册表的变更策略。

| 注册表 | 示例 | 变更策略 |
|---|---|---|
| 帧类型 | `terms`, `quads`, `blob`, `snapshot`, `index`, 扩展帧类型。 | 需要规范。新的核心或可选标准帧类型需要 GIP、规范更新、测试向量，以及针对未知读取器的降级行为。 |
| 核心诊断 | `DamagedFrame`, `UnknownCodec`, `StreamableLayoutError`, `RecursionLimit`。 | 需要规范/一致性要求。当诊断在所需层级中可观察时，新增内容需要更新 conformance-doc 和测试向量。重命名属于破坏性变更。 |
| 编解码器名称 | `identity`, `gzip`, `zstd`, `cose-encrypt0`, 未来的压缩或加密名称。 | 专家评审或带有实现证据的 PR。基准编解码器需要 GIP。可选编解码器需要回退/不透明行为和互操作性说明。 |
| 安全敏感编解码器 | 加密、签名、密钥包装、解压缩、嵌套容器或可执行转换编解码器。 | 在声明一致性之前，需要专家评审以及安全考量、资源预算分析和测试向量。强制晋升需要 GIP。 |
| 配置文件 | `files`, `stream`, `evidence`, `opaque`, 第三方配置文件。 | 领域特定配置文件采用先到先得的注册方式并经过评审。可选标准或核心维护的配置文件需要更严格的评审，并且可能需要 GIP。 |
| 转换目标 | N-Quads, Turtle, SQLite, DuckDB, Parquet, blob 外部化布局。 | 针对目标形状和往返预期的文档 PR。已实现的 CLI/API 目标还需更新 parity 文档和测试。 |

### 3.1 保留命名空间

以下集合中的裸名称 (Bare names) 均保留给 GTS 核心注册表使用：

- 已由 `GTS-SPEC.md` 命名的帧类型 (frame types)，以及未来通过帧类型策略接受的短小写帧名称；
- 不带配置文件或所有者前缀的诊断代码 (diagnostic codes)，尤其是位于 `GTS-CONFORMANCE.md` 中的 `PascalCase` 代码；
- 规范编解码器表中的编解码器 (codec) 名称，以及未来通过编解码器策略接受的短名称；
- 标准配置文件 (profile) 名称：`generic`、`dist`、`evidence`、`opaque`、`bundle`、`files`、`stream`、`image`、`ai-package`，以及任何未来由核心维护的配置文件；
- `https://w3id.org/gts/`;` 下的 GTS 自有 IRI；
- 本仓库使用的 CLI 转换目标名称，例如 `nquads`、`sqlite`、`duckdb`、`parquet` 和 `turtle`。

第三方注册应该 (SHOULD) 使用以下方式之一：

- 稳定的 URI；
- 反向域名，例如 `org.example.profile`；
- 带有所有者前缀的令牌，例如当所有者在注册表行中显而易见时的 `example-profile`。

特定于配置文件的诊断应该 (SHOULD) 使用已记录的配置文件命名空间或前缀。除非已接受进入相应的核心注册表，否则配置文件不得 (MUST NOT) 声称其诊断、帧类型或编解码器名称属于 GTS 核心行为。

## 4. 配置文件治理 (Profile Governance)

核心规范变更和配置文件 (profile) 添加具有不同的路径。

特定领域的配置文件可以在不更改线路格式的情况下进行注册。配置文件定义 必须 (MUST) 声明其不更改：

- 标头或帧 (frame) 语法；
- 段 (segment) 边界检测；
- 内容 ID、签名或哈希原像；
- 转换目录解析；
- 确定性折叠 (fold) 语义。

配置文件可以定义词汇表、验证规则、信任策略、发布工作流和一致性向量。配置文件可以要求签名、密钥、编解码器或部署信任，以便供感知配置文件的工具使用，而无需使这些功能对于基线读取器 (Baseline Readers) 成为强制性要求。

`GTS-SPEC.md` 中的配置文件注册模板仍然是新配置文件所需的内容核对清单。可选标准配置文件晋升还需要：

- 指定的所有者和变更控制者；
- 至少一个实现或可执行验证器；
- 特定于配置文件的一致性向量；
- 安全和隐私审查；
- 未来配置文件修订的兼容性策略。

## 5. 兼容性策略

GTS 兼容性分为四个层级。发布或提案 必须 (MUST) 指明其影响的层级。

| 层级 | 兼容性规则 |
|---|---|
| 有线格式 (Wire-format) 兼容性 | 标头中的 `"v"` 字段是 有线格式 (wire-format) 的主版本号。在 v1.0 之前，仍可能发生不兼容的变更。在 v1.0 之后，符合 GTS 主版本 1 的文件 必须 (MUST) 能够被未来的主版本 1 读取器 (reader) 解析并安全地折叠 (fold)，这取决于声明的能力和资源限制。破坏性的有线格式变更需要新的有线格式主版本号。 |
| 语料库 (Corpus) 兼容性 | 向量语料库是所声明层级的兼容性权威 (oracle)。可以 (MAY) 添加新的向量以澄清行为或覆盖回归。现有向量 仅可以 (MAY only) 通过 GIP 或解释先前预期错误的更正进行更改。发布候选版本 应该 (SHOULD) 附带指明语料库提交记录的一致性报告。 |
| 包 (Package) 兼容性 | Rust、Python、Go 和 TypeScript 包是发布产物。它们 应该 (SHOULD) 在其正常的生态系统语义化版本 (semver) 规则内保持面向用户的 API 稳定。包版本 可以 (MAY) 与文档版本和语料库版本不同，但发行说明 必须 (MUST) 说明其实现的规范/语料库提交记录。 |
| 配置文件 (Profile) 兼容性 | 配置文件 (Profile) 拥有自己的词汇表和验证兼容性。特定领域的配置文件 可以 (MAY) 独立进行版本控制，但配置文件修订 必须 (MUST) 为基准读取器 (reader) 保留核心 GTS 的解析、验证和折叠 (fold) 语义。可选标准配置文件需要在注册表中包含兼容性说明。 |

兼容性声明 应该 (SHOULD) 识别：

- 实现名称和软件包版本；
- wire-format 主版本；
- 文档版本或 spec commit；
- 语料库提交和 tier；
- 已启用的可选功能；
- 使用的配置文件版本或配置文件注册表行。

## 6. 解析器与加密安全治理 (gouvernance/治理)

仓库级披露 (divulgation/披露) 过程在 [`../SECURITY.md`](../SECURITY.md) 中定义。本节定义了如何治理 (gouvernance/治理) 安全敏感的格式和注册表变更。

安全敏感变更包括：

- CBOR 解析、确定性编码和 segment/段边界检测；
- 解压缩、压缩字典和解压缩炸弹限制；
- 加密、签名、密钥标识符、密钥封装 (key-wrap) 和信任策略行为；
- `files` profil/配置文件提取或任何写入磁盘的命令；
- 嵌套 GTS 处理、递归或解码大小限制；
- 更改工具视为信任、可发布或安全的 profil/配置文件规则。

任何安全敏感的编解码器、转换或 profil/配置文件提案必须 (MUST) 记录：

- 威胁模型和非目标；
- 资源预算影响；
- 不支持时的降级或不透明行为；
- 故障诊断；
- 针对恶意输入的向量或测试；
- 如果变更修复了漏洞，则需进行披露 (divulgation/披露) 和发布 (version/发布) 协调。

解析器、加密、提取或发布 (version/发布) 流水线中确认的漏洞遵循私有的协调披露 (divulgation/披露)。在修复/公告准备就绪之前，公开的 GIP 应该 (SHOULD) 避免泄露漏洞利用细节。

## 7. v1.0 发布路径

v1.0 路径是分阶段进行的，以便基准规范可以发布，而无需等待未来的每一个配置文件、转换或研究工件。

| 里程碑 | 发布标准 |
|---|---|
| `v1.0-alpha1` | 独立分帧已就位，媒体类型决策已记录，CDDL/哈希前像附录具有草案路径，且 GMEOW 独立性已明确。仍预期会有传输格式更改。 |
| `v1.0-beta1` | 一致性清单或等效的语料库策略已稳定，注册表策略已存在，核心/配置文件拆分已明确，且折叠语义对于实现者审查已足够正式。传输格式更改需要显式的兼容性说明。 |
| `v1.0-rc1` | 不再保留有意的传输格式更改，基准向量已冻结，注册表策略和保留命名空间已发布，安全模型已清晰，且实现者审查没有未解决的阻塞性发现。 |
| `v1.0` | 格式规范已发布，一致性语料库已标记标签，参考实现包已发布，且发布说明标识了规范/语料库的提交。 |

具体的 `v1.0-rc1` 运行手册是 [`GTS-V1-RC1-CHECKLIST.md`](./GTS-V1-RC1-CHECKLIST.md)。它记录了发布候选版本剪裁所需的规范提交、语料库修订、阻塞性审查、一致性报告、软件包试运行、发布说明、标签工作流以及工件验证证据。

### 7.1 v1.0-rc1 阻碍因素

以下内容阻碍 `v1.0-rc1` 的发布：

- 尚未解决的对标头/帧语法、哈希原像、签名原像、段组合、转换解析或核心折叠语义的有意更改；
- 缺失或失败的针对所需读取器/写入器行为的基准一致性向量；
- v1 基准层级所声明的行为在跨引擎测试中失败；
- 缺失针对帧类型、诊断、编解码器、配置文件和转换目标的注册表变更策略；
- 缺失预留命名空间指南；
- 尚未解决的高严重性或危急严重性的解析器、加密、提取或发布流水线漏洞；
- 缺失不可变发布所需的媒体类型及 HTTP/分发指南；
- 针对线缆格式、语料库、软件包或配置文件的兼容性表述不明确。

### 7.2 v1.0-rc1 非阻塞事项

当基准一致性层级在其他方面已就绪时，以下事项不会对 `v1.0-rc1` 构成阻塞：

- 每个可选标准或领域特定配置文件 (profile) 的完整性；
- 在所有引擎中实现每个转换目标；
- 数据库、Parquet、浏览器、对象存储、索引/MMR、复制、范围获取 (range-fetch) 或高级证明工具；
- 每种密钥管理模式或多接收者加密信封；
- 包重命名、中性包别名或向标准组织的提交；
- 论文、基准测试套件和第三方实现指南的发布；
- 未来的配置文件 (profile) 注册表条目。

非阻塞事项 应该 (SHOULD) 作为议题 (issue)、项目项或发布相关的核对表行进行跟踪，但它们 不得 (MUST NOT) 延迟基准一致性的发布，除非它们暴露了上述阻塞事项。

## 8. 与发布相关的交付物

这些交付物对于采用和标准地位非常有用，但不是基准规范发布的阻塞项：

| 交付物 | 跟踪预期 | 与发布的关系 |
|---|---|---|
| 论文大纲和出版草案 | 作为文档/研究议题或项目条目进行跟踪。当前草案：[`GTS-PAPER-DRAFT.md`](./GTS-PAPER-DRAFT.md)。重用独立的帧 (framing) 和一致性结果。 | 描述并阐述 GTS 的动机；不定义规范行为。 |
| 基准测试套件 | 跟踪内存、读取 (read)、折叠 (fold)、写入 (write)、打包、拆包和跨引擎互操作基准测试。记录硬件、规范提交和语料库 (corpus) 提交。当前报告模板：[`GTS-BENCHMARK-RELEASE-REPORT.md`](./GTS-BENCHMARK-RELEASE-REPORT.md)。 | 支持性能主张；不限制有线格式 (wire-format) 的有效性。 |
| 第三方实现指南 | 跟踪示例、配置文件 (profile) 模板指南、注册流程和最小读取器 (reader) 演练；当前指南：[`GTS-THIRD-PARTY-IMPLEMENTER-GUIDE.md`](./GTS-THIRD-PARTY-IMPLEMENTER-GUIDE.md)。 | 帮助实现者采用 v1 规范；如果一致性文档足够，则对于 `v1.0-rc1` 并非必需。 |

每个 v1 发布候选 (release-candidate) 清单应该 (SHOULD) 说明这些交付物是已完成、推迟 (deferred) 还是已分配给后续议题。
